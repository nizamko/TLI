قبل أن يُدخل أي متدرب بيانات، وقبل أن تُسجَّل أي حصة حضور، يجب أن يُبنى هيكل الثقة الكامل للنظام. هذه المرحلة تُنشئ الـ Monorepo بكامل طبقاته، تُركّب خادم Express مُحصَّن، تُعرِّف قاعدة البيانات الأولى، وتبني نظام المصادقة والصلاحيات من الصفر — بما يشمل JWT، RBAC، Tenant Isolation، وفصل الجنسين. كل مرحلة لاحقة في النظام تستند على ما يُبنى هنا.

ملخص المرحلة

ما ستُسلِّمه هذه المرحلة

المرحلة الأولى هي الحجر الأساسي لكامل النظام. لا توجد مرحلة واحدة من المراحل العشر تستطيع العمل دون المخرجات التي تُنتجها هنا — من هيكل الملفات، إلى اتصال قاعدة البيانات، وصولاً إلى نظام المصادقة بكامل طبقاته الأمنية.

🎯 الهدف

بناء بنية تحتية موثوقة وآمنة تُشغّل كل الوحدات اللاحقة. يشمل ذلك: Monorepo جاهز، خادم Express مُحصَّن بـ Helmet + CORS + Rate Limiting، اتصال PostgreSQL بـ Pool، Redis للـ Cache والـ Queues، ونظام JWT كامل مع Refresh Token Rotation.

📤 المخرجات الرئيسية

  • 🗂️هيكل monorepo: backend + frontend + shared + docs
  • 🔐نظام JWT كامل: access (15min) + refresh (7d)
  • 👮RBAC: 5 أدوار + permission matrix كاملة
  • 🏢Tenant Isolation: company_id مُحقون في كل Query
  • ⚥Gender Scope Middleware: فصل تلقائي حسب الجنس
  • 📋Audit Logs: تسجيل كل action مع old/new values
  • ⚛️Frontend: React + Vite + Zustand + TanStack Query

📥 المتطلبات السابقة

  • ✅لا تعتمد على أي مرحلة سابقة — هذه هي نقطة الانطلاق
  • 🛠️Node.js 20+ + npm/pnpm مثبت على البيئة
  • 🗄️PostgreSQL 16 قاعدة بيانات منشأة وجاهزة
  • 📦Redis 7 مشغّل ومتاح للاتصال
  • 📄ملف .env مهيأ بالمتغيرات اللازمة

الأساس أولاً — لأن كل شيء يبنى فوقه

البرمجيات الكبيرة لا تفشل بسبب خطأ في ميزة واحدة — بل تفشل لأن أساسها الأمني والمعماري مبني بشكل خاطئ. معهد ريادة الغد يتعامل مع بيانات حساسة: عقود مدربين، رواتب، سجلات حضور وغياب، وبيانات متدربين. كل هذا يستوجب أن تُبنى طبقات الحماية من اليوم الأول.

💡
السبب المعماري الأول: النظام متعدد الشركات (Multi-Tenant SaaS). إذا لم يُحقَن company_id في كل Query من البداية، فكل مرحلة لاحقة ستتطلب إعادة كتابة كاملة للـ Queries. التأسيس الصحيح هنا يوفر أسابيع من العمل لاحقاً.
⚙️
السبب الأمني: فصل الجنسين (Gender Scope) هو متطلب تشغيلي أساسي. موظف ذكر لا يجب أن يرى بيانات المتدربات إطلاقاً. هذا القيد يجب أن يكون في Middleware مركزي — وليس شرطاً يُنسى إضافته في كل API جديد يُكتب لاحقاً.
🔒
السبب التقني: Refresh Token Rotation ودوّامة الـ logout الصامتة من أصعب المشاكل لحلها لاحقاً. البناء الصحيح من البداية — مع تخزين التوكن في httpOnly cookie وتدوير صارم — يحمي المستخدمين من اختطاف الجلسات ويمنع session replay attacks.
🔑
بدونها: لا يوجد أمان
Zero security guarantees

كل API ستكتبه في المراحل 2–10 سيكون مكشوفاً بالكامل إذا لم تُبنَ طبقة المصادقة أولاً.

API Endpoints

نقاط الاتصال — Authentication & Permissions

ثمانية Endpoints تُشكّل العمود الفقري للمصادقة والتفويض في النظام. كل API لاحق في المراحل التالية يمر بهذه الطبقة قبل الوصول للبيانات.

🔐 مجموعة المصادقة — Authentication
POST /api/auth/login تسجيل الدخول — يُعيد access token + يضع refresh token في cookie
POST /api/auth/refresh تجديد الـ Access Token باستخدام Refresh Token من الـ cookie
POST /api/auth/logout إلغاء الجلسة — يُبطل الـ refresh token ويمسح الـ cookie
GET /api/auth/me بيانات المستخدم المسجّل حالياً مع صلاحياته
POST /api/auth/change-password تغيير كلمة المرور — يتطلب كلمة المرور الحالية + الجديدة
🔒 مجموعة الصلاحيات — Permissions
GET /api/permissions صلاحيات المستخدم الحالي لجميع الصفحات
POST /api/permission-requests إرسال طلب صلاحية جديد للمراجعة
📋 مجموعة سجل التدقيق — Audit Logs
GET /api/audit-logs سجل التدقيق الكامل — Super Admin فقط، مع pagination وفلاتر

منطق العمل

قواعد تشغيلية دقيقة لا تقبل الاستثناء

⚙️
قاعدة 1.1 — Pipeline الإجباري: كل API request يمر حتماً بـ 4 مراحل متسلسلة: authenticate → tenantIsolation → authorize → DB query with company_id filter. أي route لا تمر بهذه المراحل تعتبر ثغرة أمنية يجب إصلاحها فوراً.
🔒
قاعدة 1.2 — Rate Limiting: معدل الطلبات العام: 100 طلب / 15 دقيقة لكل IP. معدل تسجيل الدخول: 5 محاولات / 15 دقيقة لكل IP. عند تجاوز الحد يُعاد HTTP 429 Too Many Requests مع header يوضح وقت الانتهاء.
🔄
قاعدة 1.3 — Refresh Token Rotation: كل مرة يُستعمل فيها Refresh Token، يُنشأ واحد جديد ويُبطَل القديم فوراً في جدول sessions. إذا اكتُشف استخدام توكن مُبطَل (Replay Attack)، تُلغى جميع جلسات المستخدم تلقائياً.
🔑
قاعدة 1.4 — قوة كلمة المرور: الحد الأدنى: 8 أحرف، رقم واحد على الأقل، حرف كبير واحد على الأقل. كلمة المرور تُهشَّر بـ bcryptjs بـ cost factor 12 (حوالي 250ms على server عادي) — متعمد لإبطاء brute force.
🚫
قاعدة 1.5 — قفل الحساب التلقائي: عند 5 محاولات تسجيل دخول فاشلة متتالية، يُقفَل الحساب لمدة 15 دقيقة. يُخزَّن locked_until في جدول users ويُسجَّل في audit_logs. Super Admin يستطيع رفع القفل يدوياً قبل انتهاء المدة.
⚥
قاعدة 1.6 — Gender Scope الصارم: في أي Query يُعيد بيانات مستخدمين أو متدربين، يُحقَن تلقائياً: WHERE gender = current_user.gender OR current_user.role = 'super_admin'. هذا القيد يُضاف في genderScopeMiddleware وليس في كل Route منفصلاً.
📋
قاعدة 1.7 — Audit Logging شامل: كل عملية WRITE (INSERT / UPDATE / DELETE) تُسجَّل تلقائياً في audit_logs مع old_values + new_values كـ JSONB + IP + User Agent. عمليات القراءة الحساسة (جلب كلمة مرور، فتح ملف) تُسجَّل أيضاً كـ action = 'READ_SENSITIVE'.

مثال عملي: Tenant Isolation في Query

// ❌ Query غير آمنة — تُعيد بيانات جميع الشركات
const result = await db.query(
  'SELECT * FROM users WHERE role = $1',
  ['staff']
);

// ✅ Query آمنة — مع Tenant Isolation
const result = await db.query(
  `SELECT id, full_name, role, gender
   FROM users
   WHERE company_id = $1
     AND role = $2
     AND ($3 = 'super_admin' OR gender = $4)
     AND is_active = true`,
  [req.user.companyId, 'staff', req.user.role, req.user.gender]
);

المخاطر والافتراضات

ما يمكن أن يتأخر أو يُعيق التسليم

⚠️ المخاطر التقنية
  • 🔴
    تعقيد Tenant Isolation: إذا لم يُبنَ بشكل صحيح من البداية، ستحتاج كل المراحل اللاحقة إلى إعادة كتابة Queries — يُعيق المشروع بأكمله. عالي الأثر
  • 🟠
    JWT Refresh Loops: خطأ في منطق التجديد يتسبب في logout متكرر للمستخدمين. يصعب اكتشافه في بيئة التطوير ويظهر فقط في الـ Production. متوسط الأثر
  • 🟠
    Permission Matrix المعقدة: 5 أدوار × N صفحة × 5 permissions = جدول ضخم. إذا لم يُبنَ نظام الـ defaults بشكل صحيح، سيكون الـ onboarding لشركات جديدة بطيئاً. متوسط الأثر
  • 🟡
    bcrypt Performance: cost factor 12 يعني ~250ms لكل login. مقبول للمستخدمين البشريين، لكن يعطّل اختبارات الـ integration إذا لم تُستبدَل بـ cost 1 في بيئة test. منخفض الأثر
📌 الافتراضات والقرارات المسبقة
  • ✅
    UUID وليس Sequential ID: استخدام gen_random_uuid() كـ PK يمنع enumeration attacks ويدعم التوسع الأفقي لاحقاً.
  • ✅
    httpOnly Cookies للـ Refresh Token: قرار معماري نهائي — لا يُخزَّن في localStorage أبداً بسبب مخاطر XSS.
  • ✅
    Per-Company Permission Defaults: كل شركة جديدة تحصل على نسخة من permission matrix الافتراضية تستطيع تعديلها لاحقاً. هذا يتطلب seed script عند إنشاء كل company.
  • ✅
    Monorepo بـ Workspaces: يُقرَّر استخدام npm workspaces أو pnpm workspaces — يُسهّل مشاركة الـ Zod schemas بين Backend و Frontend.
  • ✅
    Database Migrations: جميع التغييرات عبر SQL migration files مرقّمة — لا يُسمح بأي تغيير مباشر في الـ Production database.
قائمة المخرجات

كيف نعرف أن المرحلة اكتملت بنجاح؟

هذه القائمة هي معيار القبول للمرحلة — يجب أن تتحقق كل بند قبل الانتقال للمرحلة الثانية.

  1. Monorepo مُنشأ ومُهيأ: هيكل backend/ + frontend/ + shared/ + docs/ موجود، ESLint + Prettier يعمل على كل الحزم، وسكريبت dev يُشغّل Frontend + Backend معاً ✅
  2. Express Server يعمل: يستجيب على GET /health بـ 200 OK مع حالة DB + Redis. Helmet + CORS + Rate Limiting + Compression فعّالة ✅
  3. Database Migrations نُفّذت: الجداول الستة (companies, users, sessions, permissions, permission_requests, audit_logs) موجودة في الـ DB مع الـ indexes الصحيحة ✅
  4. Auth Flow يعمل end-to-end: Login يُعيد access token + يضع refresh cookie. POST /auth/refresh يُجدّد التوكن ويدوّر الـ refresh. Logout يُبطل الجلسة ✅
  5. RBAC Middleware فعّال: Super Admin يصل لـ /api/audit-logs. Staff يحصل على 403 من نفس الـ endpoint. اختبارات موثّقة ✅
  6. Tenant Isolation صارم: مستخدم من Company A لا يرى بيانات Company B في أي endpoint. اختبار موثّق مع مستخدمَين من شركتين مختلفتين ✅
  7. Gender Scope يعمل: موظف ذكر يستدعي GET /api/users — يرى ذكوراً فقط. Super Admin يرى الجميع. موثّق باختبارات ✅
  8. Audit Logs تُسجَّل: بعد تغيير أي بيانات، صف جديد في audit_logs مع old_values + new_values صحيحة ✅
  9. Frontend Auth Store يعمل: Zustand store يحفظ بيانات المستخدم. Axios interceptor يُجدّد التوكن تلقائياً. Protected routes تُحوّل غير المسجّل إلى Login ✅
  10. الشاشات الثلاث مكتملة: Login + Change Password + Permission Request تعمل بـ validation فوري، رسائل خطأ عربية واضحة ✅
  11. CI/CD Pipeline أساسي: ESLint + Prettier check تعمل في pipeline. Build يكتمل بنجاح. لا توجد TypeScript errors ✅
  12. Account Lockout يعمل: 5 محاولات فاشلة تُقفل الحساب 15 دقيقة. locked_until يُحفَظ في DB ويُسجَّل في audit_logs ✅
