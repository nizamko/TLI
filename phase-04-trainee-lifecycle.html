لماذا هذه المرحلة؟

المتدرب هو محور النظام بأكمله

كل مرحلة لاحقة — الحضور، الإجازات، المخالفات، الرواتب، الشهادات — تبدأ بسؤال واحد: "من هو هذا المتدرب وما حالته الآن؟" بدون هذه المرحلة، لا يوجد subject يُسجَّل عليه الحضور، ولا entity يُحاسَب على المخالفة، ولا مستفيد يصدر له الشهادة.

💡
لماذا المرحلة 4 وليس الثالثة؟ — المرحلة الثالثة تُدير طلب الالتحاق وليس المتدرب نفسه. الفصل بين applications وtrainees متعمد: يسمح بحفظ سجلات المتقدمين المرفوضين دون تلويث بيانات المتدربين النشطين، ويُتيح إعادة تقديم طلب مستقبلاً بشكل منفصل.

🔄 سير الحالات — لماذا محدود؟

تحديد مسارات الانتقال المسموح بها يمنع تلاعب البيانات. مثلاً: لا يمكن إعادة تفعيل متدرب مكتمل (completed) لأن ذلك قد يُتيح له تسجيل حضور بأثر رجعي أو الحصول على إجازات جديدة بشكل غير مشروع.

📋 طلبات التعديل — لماذا workflow؟

حقول مثل الاسم والرقم الوطني تؤثر على العقود والشهادات. اشتراط موافقة الإدارة يحمي من تزوير البيانات الرسمية، بينما السماح للمتدرب بتحديث رقم هاتفه مباشرة يُقلل العبء الإداري.

تسع قواعد تشغيلية دقيقة

⚙️
قاعدة 4.1 — Status Transitions محدودة: الانتقالات المسموح بها فقط: active → suspended | active → withdrawn | active → expelled | suspended → active | suspended → expelled | active → completed (تلقائي عند التخرج فقط). حالة completed نهائية ولا يمكن الانتقال منها. أي محاولة لانتقال غير مسموح تُعيد 400 Invalid transition. 
// status-machine.js — مصفوفة الانتقالات المسموحة
const ALLOWED_TRANSITIONS = {
  active:    ['suspended', 'withdrawn', 'expelled', 'completed'],
  suspended: ['active', 'expelled'],
  completed: [],   // نهائي — لا رجوع
  withdrawn: [],   // نهائي
  expelled:  [],   // نهائي
};

function validateTransition(current, next) {
  if (!ALLOWED_TRANSITIONS[current]?.includes(next)) {
    throw new AppError(`Invalid transition: ${current} → ${next}`, 400);
  }
}
🚫
قاعدة 4.2 — Expulsion → blocked_profiles: عند استدعاء POST /api/trainees/:id/expel، يُنفَّذ في transaction واحدة: (1) UPDATE trainees SET status = expelled, (2) INSERT INTO blocked_profiles (national_id, reason, blocked_by, source = 'expulsion'). هذا يمنع المتدرب المطرود من إعادة التسجيل في أي مجموعة مستقبلاً (يتحقق منه Phase 03 عند قبول الطلب).
✏️
قاعدة 4.3 — Update Requests: حقول مباشرة vs. حقول بموافقة:
مباشرة (auto-approved): phone, emergency_contact_name, emergency_contact_phone, address.
تحتاج موافقة: national_id, national_id_expiry_date, date_of_birth, gender, photo_url, وأي حقل غير مُدرج في قائمة المباشرة.
الفصل محدد في config ثابت (لا في DB) لتسهيل التعديل دون migration.
⏰
قاعدة 4.4 — national_id_expiry_date cron: BullMQ repeatable job يعمل يومياً عند 08:00 AM. يستعلم: SELECT * FROM trainees WHERE status='active' AND national_id_expiry_date BETWEEN NOW() AND NOW()+INTERVAL '60 days'. يُرسل إشعاراً واحداً لكل نقطة: 60 يوم / 30 يوم / 7 أيام. يحفظ في notification_log لمنع التكرار. 
// cron-id-expiry.js — BullMQ daily job
const thresholds = [60, 30, 7]; // أيام

for (const days of thresholds) {
  const trainees = await queryExpiring(days);
  for (const t of trainees) {
    const alreadySent = await checkNotificationLog(t.id, `id_expiry_${days}d`);
    if (!alreadySent) {
      await sendWhatsApp(t.phone, `id_expiry`, { days });
      await logNotification(t.id, `id_expiry_${days}d`);
    }
  }
}
🔁
قاعدة 4.5 — Group Transfer وسجلات الحضور: عند نقل المتدرب إلى مجموعة جديدة، لا تُنقل سجلات attendance_records السابقة (تبقى مرتبطة بـ group_id القديم). يُنشئ النظام "نقطة بداية" جديدة للمتدرب في المجموعة الجديدة من تاريخ النقل. نسبة الحضور تُحسب من تاريخ الانضمام الفعلي للمجموعة.
🎓
قاعدة 4.6 — Graduation Threshold: عند اكتمال مجموعة (group.status = completed)، يُطلق BullMQ job لكل متدرب في المجموعة. الشرط: attendance_percentage >= 75% (قابل للتهيئة per-program في master_data). من يقل عن الحد لا يتخرج تلقائياً — يبقى active ويُنبَّه المسؤول لاتخاذ قرار يدوي.
🔒
قاعدة 4.7 — Soft Delete فقط: المتدربون ذوو الحالات النهائية (withdrawn, expelled, completed) لا يُحذفون مطلقاً من قاعدة البيانات. جميع بياناتهم محفوظة للمراجعة القانونية والإحصاءات التاريخية. الحذف الفعلي غير متاح حتى للـ Super Admin.
📂
قاعدة 4.8 — Document Upload Validation: قبل حفظ الملف: (1) التحقق من MIME type المسموح به (image/jpeg, image/png, application/pdf), (2) حجم أقصى 5MB — رفض فوري إن تجاوز, (3) ClamAV virus scan عبر BullMQ job قبل تعيين is_verified. الملف يُخزَّن في S3 بمسار خاص: trainees/{trainee_id}/docs/{uuid}.ext. 
// certificate-generator.js — قاعدة 4.9
async function generateCertificateNumber(traineeId) {
  const year = new Date().getFullYear();
  const seq  = await db.query(
    `SELECT COUNT(*)+1 AS n FROM trainees
     WHERE graduation_date BETWEEN $1 AND $2`,
    [`${year}-01-01`, `${year}-12-31`]
  );
  const padded = String(seq.rows[0].n).padStart(4, '0');
  return `RG-${year}-${padded}`; // مثلاً: RG-2026-0042
}

📥 تعتمد على

Phase 01 — الأساس: نظام المستخدمين والأدوار والـ middleware الكاملة. كل تحول حالة يسجّل changed_by من users.

Phase 02 — البيانات الرئيسية: جداول companies, departments, training_programs, groups موجودة ومُعبَّأة قبل إنشاء أي متدرب.

Phase 03 — التسجيل والقبول: الموافقة على application تُطلق إنشاء سجل المتدرب تلقائياً. blocked_profiles يُرجع إليه عند الطرد.

📤 تُسلِّم إلى

Phase 05 — الحضور: trainee_id + group_id الحالي + enrollment_date — أساس كل سجل حضور.

Phase 06 — الإجازات: status المتدرب — الإجازات لا تُقبل إن كان المتدرب withdrawn أو expelled.

Phase 07 — المخالفات: trainee_id وحالة الملف — المخالفة من degree 5 تستدعي expel API هنا.

Phase 08 — الإشعارات: phone, email لإرسال الإشعارات على تحولات الحالة وتجديد الهوية.

Phase 09 — بوابة المتدرب: الملف الشخصي + update_requests + documents — المتدرب يراها ويُعدِّل منها.

الاعتماديات الخارجية

☁️ S3 / Object Storage

تخزين صور الهوية والعقود والوثائق. يجب تهيئة bucket بـ server-side encryption ومنع الوصول العام.

🦠 ClamAV

فحص الفيروسات لكل ملف مرفوع. يعمل كـ BullMQ job منفصل بعد الرفع المؤقت قبل التثبيت الدائم.

⏰ BullMQ Scheduler

Repeatable job لـ cron الهوية المنتهية (يومياً) وجداول تخريج المجموعات (عند الاكتمال).

المخاطر والافتراضات

ما يمكن أن يتأخر أو يفشل

⚠️ المخاطر
  • 🔴تعقيد Status Transitions: أي خطأ في التحقق قد يترك المتدرب في حالة غير متسقة. الحل: unit tests لكل مسار انتقال مسموح + غير مسموح.
  • 🔴Group Transfer mid-course: هل تتأثر نسبة الحضور؟ الحل: حساب الحضور لكل group_id مستقل + تجميع عند الحاجة للتقرير الكلي.
  • 🟡Storage Cost: كل متدرب قد يرفع 4-5 وثائق. تكلفة S3 قد تتراكم. الحل: lifecycle policy لحذف وثائق المتدربين المنتهين بعد 5 سنوات.
  • 🟡Virus Scan Latency: ClamAV قد يتأخر. يجب ألا يبلوك رفع الملف — الملف يُخزَّن مؤقتاً حتى انتهاء الفحص (async).
  • 🟡Certificate Number تعارض: في حالة تخريج متزامن. الحل: استخدام SERIAL أو SEQUENCE في PostgreSQL لضمان التفرد.

قائمة المخرجات

  1. Backend — CRUD: جميع endpoints CRUD للمتدرب تعمل مع validation كامل بـ Zod، بما يشمل رفع الصورة وتخزينها في S3. ✅
  2. Backend — Status Machine: جميع مسارات الانتقال المسموحة وغير المسموحة مُختبرة بـ unit tests. كل تحول يُسجَّل في status_history. ✅
  3. Backend — Transfer: نقل المتدرب يُحدِّث group_id ويُسجَّل في group_transfers. سجلات الحضور السابقة غير مُتأثرة. ✅
  4. Backend — Documents: رفع الملفات مع التحقق من النوع والحجم و virus scan async. soft delete يعمل. ✅
  5. Backend — Update Requests: الحقول المباشرة تُطبَّق فوراً. الحقول بالموافقة تبقى pending حتى قرار admin. ✅
  6. Backend — Graduation: BullMQ job يُطلَق عند اكتمال المجموعة. certificate_number يُولَّد بتنسيق RG-YYYY-####. ✅
  7. Backend — Expulsion: طرد المتدرب يُنفِّذ INSERT في blocked_profiles في transaction واحدة. ✅
  8. Backend — ID Expiry Cron: BullMQ repeatable job يعمل يومياً ويُرسل إشعارات 60/30/7 أيام دون تكرار. ✅
  9. Frontend — S-4.1: قائمة المتدربين مع فلاتر متعددة وتصدير Excel وعرض الحالة بـ badge ملوّن. ✅
  10. Frontend — S-4.2: بطاقة المتدرب بأربع تبويبات كاملة مع أزرار تغيير الحالة + تأكيد + إدخال سبب. ✅
  11. Frontend — S-4.3 إلى S-4.6: مودال النقل، لوحة طلبات التعديل، خزانة الوثائق، شاشة تخرج المجموعة — جميعها مكتملة وتعمل. ✅
  12. Database: الجداول الخمسة مُهاجَرة مع الـ constraints والـ indexes الصحيحة (index على status, group_id, national_id_expiry_date). ✅
  13. Integration Testing: سيناريو كامل من قبول الطلب (Phase 03) → إنشاء المتدرب → نقل → تعليق → إعادة تفعيل → تخرج. ✅
  14. Documentation: Postman collection مُصدَّرة لجميع endpoints. README يصف status machine بالرسم. ✅
