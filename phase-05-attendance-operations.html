ملخص المرحلة

ما ستُسلِّمه هذه المرحلة

المرحلة الخامسة تبني نظام الحضور الكامل — من لحظة وصول إشارة البصمة من الجهاز وحتى ظهور تقرير الغياب الشهري. تشمل أربعة مصادر مُرتَّبة بأولويات صارمة، آلية مصالحة تلقائية بين المصادر المتعارضة، تقويم الإجازات الرسمية والاستثنائية، وعداد غيابات يُطلق إشعارات تلقائية عند عتبات محددة.

🎯 الهدف

بناء بنية تحتية موثوقة لتسجيل الحضور تدعم أربعة مصادر بيانات متزامنة، مع ضمان سلامة البيانات عبر آلية الأولويات والمصالحة، وتوليد إحصاءات يومية/أسبوعية/شهرية تُغذّي التقارير والإشعارات التلقائية.

📤 المخرجات

  • 6 جداول قاعدة بيانات مُهيَّأة ومُفهرَسة
  • 15 API endpoint موثَّقة ومختبَرة
  • 6 شاشات إدارية كاملة
  • BullMQ queue لمصالحة البصمة
  • Materialized view للغيابات المتراكمة
  • منطق تقويم موحَّد (رسمي + استثنائي)

📥 المتطلبات السابقة

  • المرحلة ١ — Auth، roles، JWT، database setup
  • المرحلة ٢ — Groups، rooms، schedules، training_programs
  • المرحلة ٤ — Trainees في المجموعات (trainee_id FK)
  • جهاز بصمة أو API للاختبار (mock مقبول)

🔗 يُسلِّم إلى

المرحلة ٦: الإجازات تُعدِّل سجلات الحضور (excused بدل absent) — تحتاج attendance_id.
المرحلة ٧: عداد الغيابات يُطلق درجة المخالفة الخامسة (طرد محتمل).
المرحلة ٨: تقارير الحضور الشهرية والتنبيهات المُجمَّعة تستهلك stats API.

لماذا هذه المرحلة؟

الحضور هو مركز ثقل العملية التدريبية كلها

لا يمكن بناء نظام مخالفات (مرحلة ٧) دون عداد غيابات دقيق. لا يمكن حساب الراتب (مرحلة ٧) دون معدّل حضور. لا يمكن إصدار تقارير الأداء (مرحلة ٨) دون بيانات يومية منظمة. هذه المرحلة تأتي الآن لأن البنية التحتية (مرحلة ١) والمجموعات (مرحلة ٢) والمتدربين (مرحلة ٤) جاهزة — وهي الشروط الأساسية لأي سجل حضور صالح.

📡
تعدد المصادر — مشكلة حقيقية
لماذا نحتاج أولويات صارمة؟

في الواقع العملي قد يُسجِّل المتدرب بصمة الدخول ثم يُسجِّل المشرف دخوله يدوياً في نفس اليوم. لو قبلنا المصدرين فسنحصل على سجلَّين لنفس اليوم — بيانات مكررة وأوقات متعارضة. الحل: قاعدة الأولوية تُبقي مصدر البصمة (90) وتُؤرشف المدخل اليدوي (50) في سجل التدقيق.

📅
التقويم الديناميكي — ضرورة لا رفاهية
لماذا نحتاج attendance_day_config؟

مجموعة قد تحضر أحد-ثلاثاء-خميس فقط (3 أيام/أسبوع)، ومجموعة أخرى اثنين-أربعاء-خميس. لو احتسبنا غياب السبت على مجموعة لا تتدرب فيه = بيانات خاطئة تُشعل إشعارات كاذبة وتُعطّل المراحل اللاحقة.

⚠️
ترتيب التنفيذ حرج: يجب اكتمال attendance_day_config لكل مجموعة قبل تشغيل أي عملية استيراد أو مصالحة، وإلا سيُعامَل كل يوم يوم تدريب بشكل افتراضي وتتضخم أعداد الغياب بشكل خاطئ.

الصفحات والشاشات

ست شاشات إدارية لإدارة الحضور

# الشاشة Route الوصف الأدوار
١ لوحة الحضور اليومي /attendance/daily جدول شامل بحضور كل متدرب في اليوم الحالي — فلاتر بالمجموعة والحالة، مع ملخص إحصائي سريع (نسبة الحضور، عدد المتأخرين، عدد الغائبين) Manager Staff
٢ تسجيل حضور يدوي /attendance/manual نموذج لتسجيل أو تعديل حضور متدرب محدد: اختيار التاريخ، وقت الدخول/الخروج، الحالة، ملاحظة. يُسجَّل المصدر تلقائياً كـ manual مع recorded_by Manager Staff
٣ استيراد من Excel /attendance/import رفع ملف Excel بحضور تاريخي، preview قبل الحفظ، تقرير أخطاء قابل للتنزيل (الصفوف الفاشلة مع سبب الفشل)، دعم التواريخ الهجرية والميلادية Manager
٤ استثناءات الجدول /attendance/exceptions إدارة استثناءات أيام التدريب: إلغاء يوم، نقله لتاريخ آخر، أو إضافة يوم طارئ. يؤثر على حسابات الغياب فوراً بعد الحفظ Manager
٥ الإجازات الرسمية /attendance/holidays تقويم الإجازات الوطنية والمؤسسية. إضافة، تعديل، حذف. يمكن تحديد ما إذا كانت الإجازة تؤثر على جميع المجموعات أم مجموعات محددة Manager Super Admin
٦ إعدادات أيام التدريب /attendance/day-config/:groupId تحديد أيام الأسبوع التي تتدرب فيها كل مجموعة (أحد=0 إلى سبت=6). واجهة checkbox بسيطة تُغذّي جدول attendance_day_config مباشرة Manager Staff
📱
بوابة المتدرب: يرى المتدرب سجل حضوره الشخصي (read-only) في بوابته الذاتية عبر المرحلة ٩. التسجيل الإلكتروني عبر POST /api/attendance/electronic-checkin يُتيح له تسجيل دخوله بنفسه بشرط التحقق من الموقع الجغرافي وبصمة الجهاز.

# العلاقات الأساسية trainees 1:N attendance # متدرب → سجلات حضوره groups 1:N attendance # مجموعة → حضور كل أعضائها attendance 1:N attendance_sources_log # سجل حضور → مصادر بديلة مُؤرشفة groups 1:N schedule_exceptions # مجموعة → استثناءات جدولها groups 1:7 attendance_day_config # مجموعة → 7 صفوف (أيام الأسبوع) trainees 1:1 absence_counters # متدرب → عداد غياباته (mat view) public_holidays N/A # جدول مستقل، يُستعلَم عند حساب أيام التدريب # القيود المهمة attendance: UNIQUE (trainee_id, date) # ← يمنع سجلين لنفس اليوم attendance_day_config: UNIQUE (group_id, day_of_week) # ← 7 صفوف فقط/مجموعة public_holidays: UNIQUE (date) # ← إجازة واحدة/تاريخ

API Endpoints

١٥ نقطة اتصال موزَّعة على ٤ مجموعات

📋 الحضور الأساسي — Core Attendance
GET /api/attendance قائمة الحضور مع فلاتر: group_id، trainee_id، date_from، date_to، status، source — مع pagination
POST /api/attendance إدخال يدوي — يُسجَّل source=manual تلقائياً، يُطلق trigger تحديث absence_counters
PUT /api/attendance/:id تعديل سجل حضور — المصدر الأعلى أولوية فقط يمكنه التعديل، وإلا يُرفَض الطلب
POST /api/attendance/fingerprint-sync Webhook من جهاز البصمة — يُضاف إلى BullMQ queue للمعالجة غير المتزامنة. يقبل batch من الأحداث
POST /api/attendance/electronic-checkin تسجيل دخول المتدرب عبر البوابة — يتحقق من GPS coordinates + device fingerprint قبل القبول
POST /api/attendance/excel-import رفع ملف Excel — يُعيد preview + error_report (صفوف فاشلة) قبل الحفظ الفعلي. يقبل multipart/form-data
POST /api/attendance/reconcile تشغيل المصالحة يدوياً من الإدارة — يقارن كل مصادر اليوم المحدد ويُطبِّق قاعدة الأولوية
📊 الإحصاءات والتقارير — Statistics & Reports
GET /api/attendance/summary ملخص إجمالي — يقبل فلاتر group_id، trainee_id، date_from، date_to. يُعيد: نسبة الحضور، إجمالي الحاضرين/الغائبين/المتأخرين
GET /api/attendance/stats/daily إحصاءات اليوم الحالي لمجموعة — تُغذّي لوحة القيادة. مُخزَّنة في Redis cache (ttl: 300s)
GET /api/attendance/stats/group/:id تقرير شهري تفصيلي لمجموعة — نسب الحضور لكل متدرب، الأيام المحتسبة، الغياب، التأخر
🗓️ التقويم والاستثناءات — Calendar Management
GET /api/public-holidays قائمة الإجازات الرسمية — فلتر بالسنة. يُعيد is_national و affects_all_groups
POST /api/public-holidays إضافة إجازة رسمية — يتحقق من UNIQUE(date) قبل الحفظ
PUT /api/public-holidays/:id تعديل إجازة — تغيير الاسم أو النوع أو نطاق التأثير
DELETE /api/public-holidays/:id حذف إجازة — يُعيد حساب سجلات الحضور المُؤثَّرة
GET /api/schedule-exceptions استثناءات جدول مجموعة — فلتر بـ group_id والفترة الزمنية
POST /api/schedule-exceptions إنشاء استثناء: cancelled / moved / emergency_day. لـ moved يجب إرسال original_date و new_date
PUT /api/schedule-exceptions/:id تعديل استثناء قائم
DELETE /api/schedule-exceptions/:id إلغاء الاستثناء — يُعاد احتساب الغياب للمجموعة في ذلك اليوم
⚙️ إعداد أيام التدريب — Attendance Day Config
GET /api/attendance-day-config/:group_id إعداد أيام التدريب لمجموعة — يُعيد مصفوفة 7 عناصر (0–6) مع is_training_day لكل يوم
POST /api/attendance-day-config/:group_id إنشاء الإعداد الأولي لمجموعة جديدة — يُنشئ 7 صفوف دفعة واحدة
PUT /api/attendance-day-config/:group_id تحديث أيام التدريب — يقبل مصفوفة JSON بالأيام المُفعَّلة. يُبطل Redis cache للمجموعة

١١ قاعدة تشغيلية دقيقة

⚙️
قاعدة 5.1 — أولوية المصادر: fingerprint (90) > electronic (70) > manual (50) > excel (30). عند تعارض مصدرَين لنفس اليوم: المصدر ذو الأولوية الأعلى يُحفظ في attendance، والآخر يُؤرشَف في attendance_sources_log مع priority_override=true. لا يُحذف أي مصدر أبداً — فقط يُؤرشَف للتدقيق.

قاعدة 5.2 — حساب التأخير: وقت الدخول المُسجَّل يُقارَن بـ schedule.start_time للمجموعة. تأخير > 15 دقيقة → status=late. تأخير > 60 دقيقة → status=absent (رغم حضور البصمة). نصف اليوم: خروج قبل منتصف وقت الجلسة → status=half_day. 
// Rule 5.2 — Late/Absent threshold logic
function resolveStatus(checkIn, scheduleStart) {
  const delayMinutes = diffMinutes(checkIn, scheduleStart);
  if (delayMinutes <= 15)  return 'present';
  if (delayMinutes <= 60)  return 'late';
  return 'absent';  // > 60 دقيقة
}
🗓️
قاعدة 5.3 — الإجازات الرسمية: إذا كان التاريخ موجوداً في public_holidays وكانت affects_all_groups=true (أو المجموعة ضمن القائمة المُحددة) → لا يُحتسَب هذا اليوم في الغياب. سجل الحضور يبقى null (لا يُنشأ) أو يُعلَّم كـ excused.
❌
قاعدة 5.4 — يوم ملغي: إذا وُجد سجل في schedule_exceptions لهذه المجموعة وهذا التاريخ مع exception_type=cancelled → اليوم لا يُحتسَب تدريباً، ولا يُحسَب غياب على أحد.
🔀
قاعدة 5.5 — يوم منقول: exception_type=moved → original_date يُلغى تماماً (مثل قاعدة 5.4)، وnew_date يُعامَل كيوم تدريب إضافي حتى لو لم يكن ضمن attendance_day_config الاعتيادي. الغياب في new_date يُحسَب بشكل طبيعي.
📆
قاعدة 5.6 — أيام التدريب الديناميكية: قبل احتساب أي غياب يتحقق النظام من attendance_day_config للمجموعة. إذا كان is_training_day=false لذلك اليوم → يُتجاهَل تماماً. مثال: مجموعة تتدرب أحد+ثلاثاء+خميس فقط (is_training_day=true للأيام 0,2,4 و false للبقية). 
// Rule 5.6 — isTrainingDay check
async function isTrainingDay(groupId, date) {
  const dayOfWeek = new Date(date).getDay(); // 0=Sunday
  const config = await db.query(
    `SELECT is_training_day FROM attendance_day_config
     WHERE group_id=$1 AND day_of_week=$2`,
    [groupId, dayOfWeek]
  );
  if (!config.rows[0]?.is_training_day) return false;
  const holiday = await checkPublicHoliday(date, groupId); // Rule 5.3
  const exception = await checkScheduleException(groupId, date); // Rules 5.4/5.5
  return !holiday && exception?.type !== 'cancelled';
}
🔔
قاعدة 5.7 — إشعار الغياب الأول (5 غيابات): عند تحديث absence_counters إذا وصل cumulative_absences = 5 وكان warning_level = 0 → يُرسَل إشعار WhatsApp + Email للمتدرب وللإدارة، ويُحدَّث warning_level = 1 و last_notification_sent_at. لا يُعاد الإشعار إذا كان warning_level >= 1 بالفعل.
🚨
قاعدة 5.8 — إشعار الطرد المحتمل (15 غياب): cumulative_absences = 15 → إشعار "طرد محتمل" للمتدرب + الإدارة، يُحدَّث warning_level = 2، ويُنشأ تلقائياً violation في المرحلة ٧ بدرجة 5 (أعلى درجة مخالفة). هذا الـ trigger يعمل عبر BullMQ job لا عبر HTTP request مباشر. 
// Rule 5.7 + 5.8 — Absence counter trigger (PostgreSQL function)
CREATE OR REPLACE FUNCTION check_absence_threshold()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.status = 'absent' THEN
    REFRESH MATERIALIZED VIEW CONCURRENTLY absence_counters;
    -- BullMQ job للتحقق من العتبات وإرسال الإشعارات
    PERFORM pg_notify('absence_threshold_check', NEW.trainee_id::text);
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
📍
قاعدة 5.9 — التسجيل الإلكتروني بالموقع: POST /api/attendance/electronic-checkin يتطلب: latitude، longitude، device_fingerprint. يتحقق Server من Geofence (مسافة ≤ 200 متر من الموقع المُعيَّن للمعهد). إذا خرج المتدرب عن النطاق → يُرفَض الطلب برسالة خطأ واضحة. الـ device_fingerprint يمنع تسجيل شخص آخر بدلاً عنه.
🔄
قاعدة 5.10 — Fingerprint Retry Queue: لو كان جهاز البصمة offline (timeout أو connection refused) → يُخزَّن الحدث في BullMQ مع attempts: 5 و backoff: exponential. عند استعادة الاتصال تُعالَج الأحداث المُؤجَّلة بالترتيب الزمني. الحد الأقصى لتأخير المعالجة: 24 ساعة — بعدها يُعلَّم الحدث كـ failed ويُنبَّه المشرف.
📄
قاعدة 5.11 — تقرير أخطاء Excel: كل صف فاشل في الاستيراد يُوثَّق بـ: رقم الصف، قيمة الحقل الفاشل، سبب الفشل (مثال: "رقم المتدرب غير موجود"، "تنسيق التاريخ غير صالح"، "المجموعة لا تتدرب في هذا اليوم"). التقرير يُنزَّل كـ CSV لإصلاحه وإعادة الرفع.

المخاطر والافتراضات

ما يمكن أن يتأخر أو يُعقِّد التنفيذ

⚠️ المخاطر
  • أعطال جهاز البصمة: الجهاز offline أو انقطاع الشبكة يُضيِّع أحداث دخول. الحل: BullMQ retry + تخزين محلي في الجهاز لو كان يدعم ذلك.
  • استيراد Excel — تحليل التواريخ: ملفات Excel الموروثة قد تحتوي تواريخ هجرية، أو تنسيقات مختلطة (DD/MM/YYYY vs MM/DD/YYYY)، أو تواريخ مُخزَّنة كأرقام (Excel serial). يجب بناء parser قوي يتعامل مع الحالات الثلاث.
  • Double-counting في المصالحة: لو شغّلنا reconcile مرتين على نفس اليوم → ممكن نُؤرشف سجلاً صحيحاً بالخطأ. الحل: idempotency check قبل كل مصالحة.
  • Materialized View تأخر التحديث: REFRESH CONCURRENTLY على attendance_counters بعد كل سجل غياب قد يُبطئ النظام عند الذروة (مثل نهاية الأسبوع). الحل: batch refresh كل 5 دقائق + immediate notification trigger مستقل.
  • Geofence دقة GPS: الهواتف داخل المباني قد تُعطي قراءات GPS غير دقيقة. الحل: رفع نطاق الـ Geofence لـ 500 متر مع إمكانية تعديله من الإدارة.
📌 الافتراضات
  • كل مجموعة لها attendance_day_config مُعبَّأ قبل تفعيل الحضور
  • جهاز البصمة يدعم Webhook HTTP أو API للمزامنة (ليس SFTP فقط)
  • ساعة الجهاز مُزامَنة مع NTP (±2 دقيقة كحد أقصى)
  • قاعدة البيانات تدعم pg_notify للـ triggers (PostgreSQL ≥ 12)
  • Redis متاح ومُهيَّأ مع BullMQ من المرحلة الأولى
  • Excel template موحَّد ومُوثَّق للمستخدمين المستوردين
  • إعدادات Geofence (إحداثيات المعهد) مُدخَّلة في متغيرات البيئة
🧪 حالات الحافة الحرجة
  • متدرب يُسجِّل بصمة دخول ولا يُسجِّل خروجاً → check_out_time=null، الحالة تُحسَب بناءً على check_in فقط
  • يوم منقول (moved) يقع على إجازة رسمية → الإجازة تُقدَّم على الاستثناء
  • استيراد Excel لمتدرب تسرَّب من المجموعة → يُرفَض الصف برسالة "المتدرب غير نشط"
  • نفس المتدرب يُسجِّل بصمة مرتين في نفس اليوم (دخول صباحي + دخول عودة من استراحة) → النظام يأخذ أول بصمة كـ check_in وآخر بصمة كـ check_out
