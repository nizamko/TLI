العمود الفقري لكامل النظام — هذه المرحلة تبني الهياكل التي يرتكز عليها كل شيء لاحق. من الشركة الشريكة وعقدها، إلى القسم والبرنامج التدريبي، وصولاً إلى المجموعة والقاعة. بدون هذه البيانات لا يمكن تسجيل متدرب واحد، ولا تتبُّع حضور، ولا إصدار مخالفة. كما تتضمن منطق إنهاء العقد المبكر مع cascade كامل، وتجديد جماعي لعقود الشركات.

ملخص المرحلة

ما ستُسلِّمه هذه المرحلة

المرحلة الثانية تحوِّل البنية التقنية الفارغة من المرحلة الأولى إلى منظومة بيانات حية — شركات بعقودها، أقسامها، برامجها، مجموعاتها، قاعاتها، وموظفيها. إنها "قاموس" المعهد الكامل الذي تستهلكه جميع المراحل التالية.

🎯 الهدف

بناء CRUD كامل وصحيح لكل الكيانات الرئيسية في النظام، مع تطبيق قواعد العمل الجوهرية كـ Cascade Soft-Delete عند إنهاء عقد شركة، وفرض gender restriction من البرنامج على المجموعة، وحماية تعارض حجوزات القاعات.

لماذا هذه المرحلة؟

الهيكل قبل الحركة — لماذا تأتي البيانات الرئيسية أولاً؟

النظام يعمل بتسلسل هرمي صارم: Company → Department → Training Program → Group → Trainee. أي عملية في النظام — تسجيل متدرب، تتبُّع حضور، إصدار مخالفة، دفع راتب — تحتاج أن تُشير إلى عنصر في هذه الشجرة. إذا بدأنا بـ Phase 03 (التسجيل) دون أن توجد برامج ومجموعات، ستنكسر كل عملية.

🏢 الشركة كأساس العقد

المعهد يعمل بنموذج B2B — كل متدرب يأتي من شركة شريكة بعقد محدد المدة. بدون كيان company مع contract_start_date وcontract_end_date لا يمكن حساب صلاحية تسجيل أي متدرب.

📋 البرنامج كـ Template

training_program هو القالب الثابت (مدة، أيام، سعر، ظرفية). أما group فهو "النسخة" التي تُفتح في تواريخ محددة. هذا الفصل يسمح بتكرار البرنامج عشرات المرات دون تكرار الإعداد.

🚪 القاعة كـ Resource محدود

كل group مرتبط بـ room. القاعدة 2.7 تمنع تعارض الحجوزات — لا يمكن لمجموعتين أن تشتركا في نفس القاعة في أوقات متداخلة. هذا يتطلب وجود بيانات الغرف قبل إنشاء أي مجموعة.

⚡
الترتيب الحاسم: لا يمكن إنشاء department بدون company نشطة. لا يمكن إنشاء program بدون department نشطة. لا يمكن إنشاء group بدون program. هذا التسلسل محمي بـ Foreign Key Constraints ومُعزَّز ببusiness validation في كل API.

API Endpoints

٢٧ نقطة اتصال موزَّعة على ٧ موارد

🏢 الشركات — Companies
GET /api/companies قائمة شركات مع pagination وفلتر is_active + بحث بالاسم
POST /api/companies إنشاء شركة جديدة مع validation كامل على commercial_registry
GET /api/companies/:id تفاصيل شركة مع أعداد أقسامها وبرامجها ومجموعاتها النشطة
PUT /api/companies/:id تعديل بيانات الشركة — لا يُغيِّر early_terminated_at
DELETE /api/companies/:id Soft-Delete — يُعيِّن is_active=false فقط إذا لم تكن هناك مجموعات active
POST /api/companies/:id/terminate إنهاء عقد مبكر — يحفظ reason وتاريخ التنفيذ ويُشغِّل Cascade
POST /api/companies/bulk-renew تجديد جماعي — body: { company_ids: [], extend_months: N } كل شيء في transaction واحدة
📄 عقود الشركات — Company Contracts
GET /api/companies/:companyId/contracts قائمة عقود الشركة — مرتبة بالأحدث
POST /api/companies/:companyId/contracts إنشاء عقد جديد — يتحقق أن end_date بعد start_date، ويرفع ملف العقد
PUT /api/companies/:companyId/contracts/:contractId تعديل بيانات العقد أو رفع ملف عقد بديل
DELETE /api/companies/:companyId/contracts/:contractId Soft-Delete للعقد — ممنوع لو فيه برامج تدريبية مربية عليه
📂 الأقسام — Departments
GET /api/departments قائمة أقسام — يقبل query param ?company_id للفلترة
POST /api/departments إنشاء قسم — يتحقق أن company_id يشير لشركة نشطة (Rule 2.1)
PUT /api/departments/:id تعديل بيانات القسم
DELETE /api/departments/:id Soft-Delete — is_active=false يسري على programs تبعاً
📚 البرامج التدريبية — Training Programs
GET /api/training-programs قائمة برامج — فلتر بـ department_id، is_active، gender_restriction
POST /api/training-programs إنشاء برنامج — يتحقق أن department نشطة (Rule 2.2)، كود فريد
PUT /api/training-programs/:id تعديل — تغيير duration_weeks لا يُعدِّل groups الموجودة
DELETE /api/training-programs/:id Soft-Delete — ممنوع إذا كان هناك group بحالة active
👥 المجموعات — Groups
GET /api/groups قائمة مجموعات — فلتر بـ program_id، status، trainer_id، room_id
POST /api/groups إنشاء — يحسب end_date تلقائياً (Rule 2.3)، يفحص تعارض القاعة (Rule 2.7)
PUT /api/groups/:id تعديل — تغيير room_id يُعيد فحص التعارض
DELETE /api/groups/:id Soft-Delete (status = cancelled) — ممنوع إذا كان هناك متدربون مسجَّلون
🚪 القاعات — Rooms
GET /api/rooms قائمة قاعات مع capacity وis_active
POST /api/rooms إنشاء قاعة جديدة مع قائمة المعدات (jsonb)
PUT /api/rooms/:id تعديل بيانات القاعة
DELETE /api/rooms/:id is_active=false — ممنوع إذا كانت محجوزة لمجموعة نشطة
👤 الموظفون — Employees
GET /api/employees قائمة موظفين — فلتر بـ company_id، is_active، job_title
POST /api/employees إنشاء — يتحقق أن user_id موجود وغير مرتبط بموظف آخر
PUT /api/employees/:id تعديل البيانات الوظيفية (لا يُعدِّل user_id)
DELETE /api/employees/:id is_active=false — يُلغي تعيينات المدرب النشطة إن وُجدت
📌 تعيين المدربين — Trainer Assignments
POST /api/trainer-assignments تعيين مدرب لمجموعة — يفحص التعارض الزمني للمدرب في مجموعات أخرى
GET /api/trainer-assignments قائمة تعيينات — فلتر بـ trainer_id أو group_id
DELETE /api/trainer-assignments/:id حذف تعيين — يتحقق أن المجموعة لم تبدأ بعد أو إذن خاص

منطق العمل

القواعد التشغيلية الدقيقة للمرحلة الثانية

⚙️
قاعدة 2.1 — شرط الشركة النشطة: عند إنشاء department، يجب التحقق من companies.is_active = true وأن contract_end_date > NOW(). خطأ مُقترَح: 422 COMPANY_NOT_ACTIVE.
⚙️
قاعدة 2.2 — شرط القسم النشط: عند إنشاء training_program، يجب التحقق من departments.is_active = true. القسم المعطَّل لا يستقبل برامج جديدة. خطأ مُقترَح: 422 DEPARTMENT_NOT_ACTIVE.
📅
قاعدة 2.3 — حساب end_date للمجموعة: يُحسَب تلقائياً عند إنشاء group:
end_date = start_date + (program.duration_weeks × 7 days)
المدير لا يُدخِل end_date يدوياً — تُحسَب من البرنامج.
🚪
قاعدة 2.7 — منع تعارض حجوزات القاعة (Time-based): قبل إنشاء أو تعديل group، يُنفَّذ فحص للأيام (schedule_days) والتوقيتات (start_time, end_time). يُسمح بمجموعتين في نفس القاعة في نفس اليوم فقط إذا لم يتداخل الوقت (مثال: مجموعة من 4-6 ومجموعة من 6-8).
إذا تداخل الوقت، يُفحص أن مجموع سعة المجموعات لا يتجاوز سعة القاعة الكلية. إذا تجاوز، يُرفض الطلب بـ 409 ROOM_CONFLICT. 
// Cascade Terminate — pseudo-code
async function terminateCompany(companyId, reason) {
  await db.transaction(async (trx) => {
    await trx('companies').update({
      is_active: false,
      early_terminated_at: new Date(),
      early_terminated_reason: reason
    }).where({ id: companyId });

    const deptIds = await trx('departments')
      .update({ is_active: false })
      .where({ company_id: companyId })
      .returning('id');

    const progIds = await trx('training_programs')
      .update({ is_active: false })
      .whereIn('department_id', deptIds)
      .returning('id');

    await trx('groups')
      .update({ status: 'cancelled' })
      .whereIn('program_id', progIds)
      .whereNotIn('status', ['completed']);
  });
}
⚙️
قاعدة 2.5 — وراثة قيد الجنس: عند إنشاء group، إذا كان program.gender_restriction != 'mixed'، يُجبَر group.gender = program.gender_restriction ولا يمكن للمدير تغييره.
🔄
قاعدة 2.6 — Bulk Renewal في Transaction: POST /api/companies/bulk-renew يُنفِّذ كل التحديثات في transaction واحدة. إذا فشل أي تحديث (مثلاً company محذوفة)، يُلغى الكل ويُرجَع تقرير تفصيلي بالأسباب.
🚪
قاعدة 2.7 — منع تعارض حجوزات القاعة: قبل إنشاء أو تعديل group، يُنفَّذ الاستعلام التالي. إذا أعاد أي نتيجة، يُرفَض الطلب بـ 409 ROOM_CONFLICT:
SELECT id FROM groups WHERE room_id = :room AND status NOT IN ('cancelled','completed') AND NOT (end_date < :start OR start_date > :end)
⚙️
قاعدة 2.8 — التحقق من سعة العقد (Contract Capacity): عند إنشاء training_program، يجب التحقق من أن max_capacity للبرنامج لا يتجاوز السعة المتبقية في العقد (agreed_trainees_count مطروحاً منه مجموع سعة البرامج الأخرى المربوطة بنفس العقد). خطأ مُقترَح: 422 CONTRACT_CAPACITY_EXCEEDED.

المخاطر والافتراضات

ما يمكن أن يتأخر أو يفشل

⚠️ المخاطر
  • Orphan Records: Cascade Soft-Delete إذا انقطع في المنتصف (خطأ DB) قد يُبقي بعض groups نشطة رغم تعطيل شركتها. الحل: transaction + rollback صريح.
  • Gender Restriction Complexity: عند تغيير gender_restriction للبرنامج بعد إنشاء groups، هل تُحدَّث القيود تلقائياً؟ القرار: لا تأثير على groups الموجودة، فقط الجديدة.
  • Bulk Operations Failure: تجديد 50 شركة في transaction واحدة — أي deadlock يُلغي الكل. الحل: retry logic + chunking بـ 10 شركات.
  • Room Conflict Race Condition: طلبان متزامنان لحجز نفس القاعة. الحل: SELECT ... FOR UPDATE في transaction.
  • Trainer Time Conflict: مدرب مُعيَّن لمجموعتين تتداخل أوقاتهما. يحتاج فحص صريح في trainer-assignments.
📌 الافتراضات
  • كل شركة لها commercial_registry فريد — هذا مُلزَم بـ UNIQUE constraint
  • Duration_weeks للبرنامج لا يتغير بعد إنشاء مجموعات — تغييره يؤثر فقط على المجموعات الجديدة
  • موظف لا يمكنه الانتماء لأكثر من شركة في نفس الوقت
  • شعار الشركة اختياري — النظام يعمل بدونه
  • لا يُحذَف أي سجل نهائياً في هذه المرحلة — كل الحذف Soft-Delete
  • التجديد الجماعي يُعدِّل contract_end_date فقط ولا يُعيد تفعيل شركة مُنهاة مبكراً
📐 قرارات تصميمية
  • Equipment في rooms مخزَّن كـ jsonb مصفوفة strings — مرونة دون جدول منفصل
  • gender_restriction كـ enum PostgreSQL — أفضل من varchar للـ constraint
  • trainer_assignments جدول منفصل (وليس FK مباشر) — يدعم تاريخ التعيينات والمدرب المساعد
قائمة المخرجات

كيف نعرف أن المرحلة الثانية انتهت؟

  1. قاعدة البيانات: الجداول السبعة مُنشأة بكل الـ constraints (FK, UNIQUE, NOT NULL, CHECK للـ enums). Indexes على company_id, department_id, program_id, room_id. ✅
  2. Backend — CRUD: جميع الـ 27 endpoint مُختبَرة بـ Postman Collection كاملة. Validation بـ Zod على كل route. Auth guards على كل endpoint. ✅
  3. Backend — Business Rules: Rules 2.1 إلى 2.7 مُختبَرة بـ unit tests و integration tests. Cascade terminate مُختبَر مع rollback scenario. ✅
  4. Frontend — شاشات: الشاشات السبع مبنية وتعمل مع real API (ليس mock data). Forms مُحققة بـ react-hook-form + zod client-side. ✅
  5. Frontend — UX Confirmations: Dialog تأكيد لـ Terminate و Bulk Renew يعرض ملخص الأثر قبل التنفيذ. ✅
  6. Audit Logs: كل عملية حساسة (terminate, bulk-renew, delete) مُسجَّلة في audit_logs مع المستخدم والوقت والتفاصيل. ✅
  7. Room Conflict: اختبار concurrent requests لحجز نفس القاعة — النظام يرفض الثاني بـ 409. ✅
  8. التوثيق: هذه الصفحة محدَّثة وتعكس الـ schema الفعلي المُنفَّذ. Postman Collection مُصدَّرة في المستودع. ✅
