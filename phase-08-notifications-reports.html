الإشعارات والتقارير
محرك الاتصال والتحليل الذكي

محرك إشعارات متكامل يرسل رسائل WhatsApp وEmail بـ١٧ قالباً قابلاً للتحرير، مع queue ذكي ببصمة BullMQ + Redis يضمن التسليم بإعادة المحاولة والنسب المتدرجة. ومحرك تقارير شامل يُنتج ملفات PDF و Excel و CSV عند الطلب أو وفق جداول زمنية مبرمجة — تغطي الحضور والرواتب والمخالفات والتميز وأداء البرامج. هذه المرحلة تُحوِّل البيانات المتراكمة في جميع المراحل السابقة إلى رسائل فعّالة وتقارير قابلة للتنفيذ.

ملخص المرحلة

ما ستُسلِّمه هذه المرحلة

المرحلة الثامنة تبني الطبقة التواصلية للنظام — إشعارات تلقائية تصل للمتدرب والمسؤول في اللحظة الصحيحة، وتقارير دقيقة يُصدِّرها المدير بنقرة. تُغلق هذه المرحلة حلقة البيانات: كل ما جُمع في المراحل السابقة (حضور، مخالفات، رواتب، إجازات) يتحول هنا إلى قيمة مباشرة ومعلومة قابلة للعمل.

🎯 الهدف

بناء محرك إشعارات موثوق قائم على Queue مع دعم WhatsApp و Email و In-App، ومحرك تقارير يُنتج ملفات جاهزة للطباعة والتدقيق عند الطلب أو بشكل دوري مجدول.

بدون إشعارات وتقارير النظام أصم وأعمى

المراحل السابقة (١-٧) بنت محرك البيانات: تسجيل، حضور، إجازات، مخالفات، رواتب. لكن هذه البيانات بلا قيمة إذا لم تصل إلى صاحبها في الوقت المناسب، أو إذا لم يستطع المدير العليا قراءتها في تقرير موحد. المرحلة الثامنة تبني "طبقة التواصل" التي تربط النظام بمستخدميه وتُحوِّل أحداث قاعدة البيانات إلى قرارات فعلية.

⏰
التوقيت الحرج
Critical Timing

إشعار انتهاء الهوية قبل ٦٠ يوماً يمنع التأخير. إشعار تحذير الغياب بعد ٥ مرات يُوقف الانزلاق نحو الفصل قبل فوات الأوان. الإشعار المتأخر إشعار بلا قيمة.

📊
القرار بالبيانات
Data-Driven Decisions

تقرير أداء المجموعة يكشف انخفاض الحضور قبل الإنهاء. تقرير الرواتب يُسرِّع مراجعة المحاسب. التقارير المجدولة تصل لصندوق البريد مباشرة بدون طلب.

🔁
الموثوقية
Delivery Reliability

Queue مع إعادة المحاولة يضمن وصول الإشعار حتى لو فشل مزوّد WhatsApp لدقائق. سجل كامل لكل محاولة إرسال يُجيب على "هل وصلت الرسالة؟" بدليل.

📌
لماذا الآن (المرحلة ٨)؟ لأن جميع مصادر البيانات جاهزة: الحضور (٠٥)، الإجازات (٠٦)، المخالفات والرواتب (٠٧). محرك الإشعارات يحتاج هذه المصادر ليُطلِق الأحداث الصحيحة. تأجيله لما بعد المرحلة ٩ كان سيجعل بوابة المتدرب بلا إشعارات — وهي إحدى أهم ميزاتها.

محرك الإشعارات
Notifications Engine
  • WhatsApp عبر Twilio / 360dialog Business API
  • Email عبر SendGrid أو SMTP مخصص
  • In-App (داخل النظام) للمستخدمين المسجّلين

محرك التقارير
Reports Engine
  • تقارير حضور: يومي / أسبوعي / شهري / لكل مجموعة / لكل متدرب
  • تقارير مالية: رواتب / خصومات / مكافآت

API Endpoints

١١ نقطة اتصال — الإشعارات والتقارير

🔔 الإشعارات — Notifications
GET /api/notifications إشعارات المستخدم الحالي مع pagination وفلتر الحالة والقناة
POST /api/notifications/send إرسال يدوي من Admin — يحدد: user_id، template_key، variables، channel، priority
POST /api/notifications/:id/mark-read تعليم الإشعار كـ "مقروء" — يُحدِّث read_at
POST /api/notifications/mark-all-read تعليم كل إشعارات المستخدم كمقروءة دفعةً واحدة
GET /api/notifications/stats إحصاءات Admin: sent/delivered/failed per channel لآخر ٣٠ يوماً
📝 قوالب الإشعارات — Notification Templates (Super Admin)
GET /api/notification-templates قائمة كل القوالب مع حالة التفعيل
PUT /api/notification-templates/:key تحرير قالب: subject_template، body_template، تفعيل/تعطيل
⚙️ تفضيلات المستخدم — Notification Preferences
GET /api/notification-preferences جلب تفضيلات المستخدم الحالي لكل الفئات والقنوات
PUT /api/notification-preferences تحديث تفضيلات دفعية — array من { category, channel, is_enabled }
📊 التقارير — Reports
GET /api/reports قائمة التقارير المُولَّدة مسبقاً مع حالتها وروابط التنزيل
POST /api/reports/generate توليد تقرير — sync للصغير (<١MB) | async لـ BullMQ للكبير. يُعيد report_id فوراً
GET /api/reports/:id/download تنزيل ملف التقرير — يُعيد Presigned S3 URL (صالح ١٥ دقيقة)
🗓️ التقارير المجدولة — Scheduled Reports (Super Admin)
GET /api/scheduled-reports قائمة التقارير المجدولة مع next_run_at وlast_run_at
POST /api/scheduled-reports إنشاء جدول تقرير جديد — تضمين schedule_cron + recipients array
PUT /api/scheduled-reports/:id تعديل الجدول: cron، المستلمين، الفلاتر، التفعيل
POST /api/scheduled-reports/:id/run-now تشغيل فوري للاختبار — يُدرِج مهمة في BullMQ مباشرة

قواعد تشغيلية دقيقة — ١١ قاعدة

⚙️
قاعدة 8.1 — نظام الأولوية: كل notification تُضاف لـ BullMQ Queue بـ priority: critical=1، high=2، normal=3، low=4. الـ Worker يسحب الأعلى أولوية أولاً. إشعارات كلمة المرور وتسجيل الدخول الجديد لا تُقابَل بـ User Preferences — تُرسَل دائماً.
🔁
قاعدة 8.2 — إعادة محاولة WhatsApp: فشل الإرسال → retry ٣ مرات بـ exponential backoff: 1 دقيقة، 5 دقائق، 15 دقيقة. بعد ٣ فشل → status = failed + إدراج سبب الفشل في failed_reason.
📧
قاعدة 8.3 — إعادة محاولة Email: retry ٥ مرات: 2 دقيقة، 10 دقائق، 30 دقيقة، 2 ساعة، 6 ساعات. بعد ٥ فشل → failed. Email bounce يُسجَّل في provider_response.
🙏
قاعدة 8.4 — احترام التفضيلات: قبل إضافة أي notification للـ Queue، يفحص النظام notification_preferences. إذا opt-out من category × channel → skip ولا يُسجَّل الإرسال. الإشعارات الأمنية (password_reset، new_device_login) تُتجاوز التفضيلات دائماً.
✏️
قاعدة 8.5 — متغيرات القوالب: syntax يشبه Handlebars: {'{'}{'{'}trainee_name{'}'}{'}'}، {'{'}{'{'}absences_count{'}'}{'}'}، {'{'}{'{'}date_hijri{'}'}{'}'}. القالب يُحدِّد variables_schema (JSONB) الذي يُوثِّق المتغيرات المطلوبة. Notification Service يتحقق من وجودها قبل الإرسال.
🌐
قاعدة 8.6 — دعم RTL للغة العربية: قوالب WhatsApp تستخدم أرقام هندية-عربية (٥ غيابات لا 5 absences) حيثما كانت موجَّهة للمتدرب. قوالب Email تُضمِّن dir="rtl" في HTML مع خط Tajawal. تقارير PDF تُوّلَد بـ Puppeteer + إعدادات direction: rtl كاملة.
📦
قاعدة 8.7 — التقارير الكبيرة (Async): إذا تجاوز التقرير المتوقع ١ MB أو 10,000 سطر → توليد async تلقائياً. يُعاد report_id فوراً بـ status=pending. Worker يُولِّد → يرفع S3 → يُحدِّث status=ready → يُرسِل إشعار in_app + email للمستخدم.
⏰
قاعدة 8.8 — Cron المجدوِل: schedule_cron يُحلَّل بـ croner library (يدعم 6 حقول بما فيها الثانية). next_run_at يُحسَب عند الإنشاء وبعد كل تشغيل. إذا تأخر التشغيل +٥ دقائق عن next_run_at تُسجَّل مشكلة في Winston logs.
🗑️
قاعدة 8.9 — انتهاء صلاحية الملفات: ملفات التقارير تُرفع لـ S3 بـ metadata expires_after=7days. AWS S3 lifecycle rule يحذف الملف تلقائياً. expires_at يُسجَّل في جدول reports لإخفاء الرابط من الواجهة قبل انتهاء الصلاحية.
📄
قاعدة 8.10 — PDF بالعربية: Puppeteer يُحمِّل HTML template كاملة مع CDN خط Tajawal + dir="rtl" + text-align: right + page-break-inside: avoid للجداول. الـ headless Chrome يُولِّد PDF بجودة طباعة. يُضاف header وfooter مع اسم التقرير وتاريخ التوليد وشعار المعهد.
🚫
قاعدة 8.11 — Rate Limiting: WhatsApp: max ١٠ رسائل / مستخدم / يوم (للوقاية من spam). تُتحقق عبر Redis counter بـ TTL 24 ساعة. إذا تجاوز الحد: Notification status = failed + failed_reason = "rate_limit_exceeded". يُستثنى من الحد: إشعارات critical (priority=1).

مثال على منطق الـ Queue Worker

// BullMQ Worker — processNotification(job)
const { notificationId } = job.data;
const notif = await db.query('SELECT * FROM notifications WHERE id = $1', [notificationId]);

// 1. تحقق من التفضيلات
const pref = await getUserPreference(notif.user_id, notif.template_key, notif.channel);
if (!pref.is_enabled && notif.priority > 1) {
  await updateStatus(notificationId, 'skipped');
  return;
}

// 2. تحقق من Rate Limit
if (notif.channel === 'whatsapp') {
  const count = await redis.incr(`wa:limit:${notif.user_id}`);
  if (count === 1) redis.expire(`wa:limit:${notif.user_id}`, 86400);
  if (count > 10 && notif.priority > 1) {
    await updateStatus(notificationId, 'failed', 'rate_limit_exceeded');
    return;
  }
}

// 3. استبدال المتغيرات في القالب
const template = await getTemplate(notif.template_key);
const body = renderTemplate(template.body_template, notif.variables);

// 4. الإرسال عبر المزوِّد
try {
  const result = await sendViaProvider(notif.channel, notif.recipient, body);
  await logAttempt(notificationId, job.attemptsMade, 'success', result);
  await updateStatus(notificationId, 'sent');
} catch (err) {
  await logAttempt(notificationId, job.attemptsMade, 'failed', err.message);
  throw err; // BullMQ يُعيد المحاولة تلقائياً
}

مثال على Puppeteer لتوليد PDF عربي

// generatePDF(reportType, data) → Buffer
const browser = await puppeteer.launch({ args: ['--no-sandbox'] });
const page = await browser.newPage();
const html = buildReportHTML(reportType, data); // RTL + Tajawal + Arabic numerals
await page.setContent(html, { waitUntil: 'networkidle0' });
const pdf = await page.pdf({
  format: 'A4',
  printBackground: true,
  margin: { top: '20mm', bottom: '20mm', right: '15mm', left: '15mm' },
  displayHeaderFooter: true,
  headerTemplate: `<div style="font-family:Tajawal;direction:rtl">معهد ريادة الغد</div>`,
});
await browser.close();
return pdf;

المخاطر والافتراضات

ما يمكن أن يتأخر أو يفشل

⚠️ المخاطر
  • WhatsApp Provider Downtime: Twilio أو 360dialog قد يعاني انقطاعاً. الحل: Circuit Breaker + Fallback لـ Email إذا فشل WhatsApp ٣ مرات.
  • Email Deliverability: إذا لم تُعدَّ SPF + DKIM + DMARC صحيحاً فستُحجب الرسائل كـ spam. يجب تهيئة DNS قبل الإطلاق.
  • PDF Generation Cost: Puppeteer يستهلك ذاكرة كبيرة. تقارير الشركات الكبيرة قد تستغرق 30+ ثانية. الحل: timeout=60s + memory limit per worker.
  • Template Variable Mismatch: إذا أُضيف حدث جديد بمتغير غير معرَّف في variables_schema → الإشعار يفشل بصمت. الحل: validation قبل الإضافة للـ Queue.
  • Redis Failover: إذا سقط Redis تُفقد Queue المهام المعلَّقة. الحل: Redis Cluster + AOF persistence.
  • S3 Costs: التقارير الضخمة (كل يوم) قد تتراكم. lifecycle rule 7 أيام ضروري وليس اختيارياً.
📌 الافتراضات
  • حساب WhatsApp Business API جاهز ومُوثَّق مع رقم المعهد قبل التطوير.
  • SendGrid أو SMTP مُعدَّان بـ SPF/DKIM/DMARC قبل اختبار الإشعارات.
  • AWS S3 Bucket مُنشأ مع Lifecycle Policy من البداية.
  • Redis مُشغَّل بـ persistence (AOF) في بيئة production.
  • المتدربون لديهم أرقام هاتف صحيحة في ملفاتهم (الهاتف حقل مطلوب في التسجيل).
  • كل template_key مُعرَّف في notification_templates قبل أول إرسال.
  • الخوادم لديها Chromium مُثبَّت لـ Puppeteer (أو استخدام puppeteer-core مع مسار محدد).
  • croner يعمل في process واحد — لو كانت خوادم متعددة يحتاج distributed lock (Redlock).
💡
توصية للتخفيف: في المرحلة الأولى يمكن تأجيل WhatsApp والبدء بـ Email فقط حتى يُوثَّق الحساب. مركز الإشعارات In-App يعمل فوراً بدون أي مزوِّد خارجي. التقارير PDF يمكن البدء بـ Excel فقط ثم إضافة Puppeteer لاحقاً.
