التسجيل والقبول
بوابة دخول كل متدرب للنظام

المرحلة الثالثة هي نقطة دخول كل متدرب جديد — تبدأ برابط تسجيل مُتحكَّم يُصمَّم لكل برنامج أو مجموعة، مروراً بنموذج ديناميكي قابل للبناء من قِبَل المسؤول، ثم خط مراجعة ذكي يحتوي على فحوصات تلقائية للعمر والجنس والطاقة الاستيعابية وقوائم الحظر. لا يصل أي متدرب لقاعدة البيانات إلا بعد موافقة صريحة أو اجتياز كامل للـ auto-review — عندها يُنشأ حساب وسجل تلقائياً ويُرسل إشعار WhatsApp/Email فوراً.

ملخص المرحلة

ما ستُسلِّمه هذه المرحلة

مرحلة التسجيل والقبول هي الطريق الرسمي الوحيد لإدخال أي متدرب في النظام. تُغطي كل شيء من بناء النموذج، وإنشاء رابط التسجيل، حتى المراجعة والقبول وإنشاء الحساب تلقائياً.

🎯 الهدف

توفير مسار تسجيل رقمي متكامل: نموذج ديناميكي قابل للتخصيص، روابط تسجيل مُتحكَّمة لكل برنامج أو مجموعة، ومسار مراجعة ذكي يُقلل التدخل اليدوي ويضمن دقة البيانات ومطابقة الشروط قبل القبول.

📤 المخرجات

  • ✅نظام Form Builder ديناميكي (أقسام + حقول)
  • ✅روابط تسجيل منتهية الصلاحية وبحد أقصى للاستخدام
  • ✅صفحة تسجيل عامة بدون auth تستهلك الرابط
  • ✅خط مراجعة: pending → approved / rejected / needs_info
  • ✅Auto-review rules (عمر، جنس، طاقة، حظر)
  • ✅Excel bulk import مع error report
  • ✅قائمة الملفات المحظورة (auto-reject)
  • ✅إنشاء user account + trainee record عند القبول

📥 المتطلبات السابقة

  • 🔗Phase 01: Auth system، JWT، audit_logs، rate limiting
  • 🔗Phase 02: companies، departments، training_programs، groups — كل جداول البيانات الرئيسية يجب أن تكون موجودة قبل إنشاء أي رابط تسجيل
  • 📌S3 bucket أو local storage مُعَدّ لرفع الصور
  • 📌WhatsApp API و SMTP مُعَدَّان للإشعارات

🔗 يُسلِّم إلى

Phase 04 — دورة حياة المتدرب: جدول trainees المُنشأ عند approve يصبح الكيان الرئيسي لإدارة العقود والرواتب والتعديلات. جدول users المُنشأ تلقائياً يُمكِّن تسجيل دخول المتدرب لبوابته في Phase 09.

لماذا هذه المرحلة؟

البوابة التي تحدد جودة كل ما بعدها

بدون مرحلة تسجيل مُنظَّمة، كل مرحلة تالية تعمل على بيانات غير موثوقة. الحضور لا يمكن تسجيله بدون متدرب. الرواتب لا تُحسب بدون عقد. المخالفات لا تُطبَّق بدون سجل. هذه المرحلة هي المرشِّح الأول والأهم.

🔒
تحكم كامل في من يدخل
لا تسجيل بدون رابط صالح

كل رابط تسجيل مرتبط ببرنامج محدد أو مجموعة محددة، وله تاريخ انتهاء صلاحية وعدد استخدامات أقصى. لا يمكن لأحد الوصول للنموذج العام خارج هذه القيود.

🤖
مراجعة تلقائية ذكية
قواعد auto-review تُقلل الجهد اليدوي

النظام يفحص كل طلب تلقائياً ضد قائمة الحظر، شروط العمر والجنس، والطاقة الاستيعابية للمجموعة. الطلبات المستوفية تُقبَل تلقائياً أو توضع في قائمة انتظار.

⚡
إنشاء فوري عند القبول
لا خطوات يدوية بعد approve

عند موافقة المسؤول (أو Auto-approve)، يُنشئ النظام حساب المستخدم وسجل المتدرب تلقائياً ويُرسل إشعار ترحيب عبر WhatsApp و Email — صفر تدخل يدوي.

💡
ترتيب المراحل: هذه المرحلة تأتي بعد Phase 02 مباشرة لأنها تستهلك برامج التدريب والمجموعات المُعرَّفة مسبقاً. لو حُذفت هذه المرحلة، لن يكون هناك طريقة منظمة لإدخال المتدربين — ستُفقد سيطرة المعهد على من يدخل ومن يُرفض.

الصفحات والشاشات

٦ شاشات تغطي دورة التسجيل الكاملة

# الشاشة المسار الوصف الأدوار
1 Form Builder /admin/form-builder بناء وتعديل نموذج التسجيل — أقسام وحقول ديناميكية مع معاينة حية. Super Admin فقط يقدر يعدّل هيكل النموذج. Super Admin
2 Registration Links /admin/registration-links إنشاء وإدارة روابط التسجيل. كل رابط مرتبط ببرنامج/مجموعة مع تاريخ انتهاء وحد استخدام. يعرض إحصاءات الاستخدام لكل رابط. Super Admin Manager
3 Public Registration /register/:code صفحة عامة بدون تسجيل دخول — يصل إليها المتقدم بالرابط. تعرض النموذج الديناميكي مع CAPTCHA. لا تُحتسَب الصفحة إذا كان الرابط منتهياً أو مستنفداً. عام (بدون auth)
4 Applications Review /admin/applications قائمة الطلبات مع فلاتر متقدمة. يعرض نتيجة auto_review لكل طلب. أزرار approve/reject/needs_info. إدارة Waitlist. Super Admin Manager Staff
5 Excel Bulk Import /admin/bulk-import رفع ملف Excel، معاينة البيانات قبل الاستيراد، تنزيل template فارغ، عرض نتائج الاستيراد مع error report قابل للتنزيل. Super Admin Manager
6 Blocked Profiles /admin/blocked-profiles قائمة المحظورين مع فلتر بالهوية والاسم والشركة. إضافة حظر يدوي. التمييز بين auto_blocked (من النظام) والحظر اليدوي. Super Admin Manager
📱
ملاحظة UX: صفحة التسجيل العامة (/register/:code) يجب أن تكون mobile-first بالكامل — معظم المتقدمين سيصلون إليها من رسالة WhatsApp على هواتفهم. يجب أن تعمل على اتصالات 3G ضعيفة، مع تقليص الصورة على الـ client قبل الرفع.

POST /api/public/applications PUBLIC — إرسال طلب تسجيل جديد (يستهلك الرابط + يشغّل auto-review)

٩ قواعد تشغيلية حاسمة

🔒
قاعدة 3.1 — Link Validation: قبل عرض النموذج العام، يتحقق النظام من: (1) is_active = true، (2) expires_at إما null أو أكبر من now()، (3) max_uses إما null أو current_uses أقل منه. أي فشل في أي شرط → HTTP 410 Gone مع رسالة واضحة للمتقدم.
🚫
قاعدة 3.2 — Blocked Profile Auto-Reject: عند استلام طلب جديد، يتحقق النظام فوراً من جدول blocked_profiles بـ national_id. لو موجود → status = rejected تلقائياً + auto_review_result.blocked = true + إشعار للمسؤول (لا يُرسل إشعار للمتقدم الـ reason للأمان). يُسجَّل في audit_logs.
⚥
قاعدة 3.3 — Gender Matching: إذا كان البرنامج المرتبط بالرابط يحتوي على gender_restriction (male_only أو female_only)، يجب أن يطابق جنس المتقدم هذا القيد. فشل المطابقة → rejected تلقائياً + auto_review_result.gender_match = false.
🎂
قاعدة 3.4 — Age Verification: إذا كان النموذج يحتوي على حقل date_of_birth مع validation_rules.min_age أو validation_rules.max_age، يحسب النظام العمر بالسنوات من date_of_birth لتاريخ اليوم. أي خروج عن النطاق المحدد → rejected تلقائياً. مثال: min_age = 18، max_age = 45.
📊
قاعدة 3.5 — Capacity Check & Waitlist: يتحقق النظام من عدد المتدربين في المجموعة المستهدفة. إذا كان group.current_trainees >= group.max_trainees → status = waitlisted (ليس rejected). لو أُلغي متدرب أو انسحب لاحقاً، يُرقَّى أول طلب في الـ waitlist تلقائياً.
⚡
قاعدة 3.6 — Auto-Create on Approve: عند تغيير status إلى approved (يدوياً أو تلقائياً)، تنفَّذ transaction تحتوي: (1) إنشاء user بـ username = national_id وكلمة مرور مؤقتة، (2) إنشاء trainee record مرتبط بـ application_id وـ group_id، (3) تحديث group.current_trainees++، (4) إرسال BullMQ job لـ WhatsApp + Email مع كلمة المرور المؤقتة. كل هذا في transaction واحدة — أي فشل يُلغي الكل.
🪪
قاعدة 3.7 — National ID Expiry: حقل national_id_expiry_date يُحفظ إما يدوياً من قِبَل المراجع أو بشكل تلقائي إذا رفع المتقدم صورة بطاقة. يُستخدم في Phase 08 لإرسال إشعار قبل 30 يوم من الانتهاء. لو كانت البطاقة منتهية عند التسجيل → تحذير للمراجع (ليس رفضاً تلقائياً).
📊
قاعدة 3.8 — Excel Import Error Report: عند bulk import، يُعالَج كل صف بشكل مستقل. الصفوف الفاشلة (validation error أو auto-review reject) تُجمَّع في error report بنفس هيكل الـ template + عمود إضافي "سبب الفشل" باللغتين. يُرسَل كـ .xlsx. الصفوف الناجحة تُدخَل حتى لو فشل بعضها.
❌
قاعدة 3.9 — Rejection Reason Mandatory: لا يمكن رفض طلب بدون rejection_reason غير فارغ (min 10 حروف). النظام يتحقق من هذا في backend أيضاً، ليس فقط frontend. rejection_reason يُرسَل للمتقدم عبر WhatsApp/Email حرفياً — لذا يجب أن يكون مناسباً ومهنياً.

منطق Auto-approve vs Manual Review

// التسلسل الكامل لمعالجة طلب جديد
async function processApplication(application) {
  const review = await runAutoReview(application);

  if (review.blocked_profile) {
    return setStatus('rejected', 'محظور من النظام'); // قاعدة 3.2
  }

  if (!review.gender_match || !review.age_check.pass) {
    return setStatus('rejected', buildReason(review)); // قاعدة 3.3، 3.4
  }

  if (!review.capacity.pass) {
    return setStatus('waitlisted'); // قاعدة 3.5
  }

  return setStatus('pending'); // يذهب للمراجعة اليدوية
}
خريطة التبعيات

قبل وبعد مرحلة التسجيل

📥 تعتمد على (Requires)

Phase 01 — الأساس المعماري:
نظام auth و JWT لحماية admin endpoints. جدول users لإنشاء حساب المتدرب. نظام audit_logs لتسجيل عمليات approve/reject/block. Rate limiting للحماية من spam على الصفحة العامة. BullMQ queue للإشعارات عند approve.

Phase 02 — البيانات الرئيسية:
training_programs وـ groups يجب أن تكون موجودة قبل إنشاء أي رابط تسجيل. gender_restriction في training_programs تُستخدم في auto-review. max_trainees في groups تُستخدم في capacity check. companies تُستخدم لتحديد نطاق blocked_profiles.

📤 تُسلِّم إلى (Delivers)

Phase 04 — دورة حياة المتدرب:
جدول trainees المُنشأ عند approve يحمل كل التفاصيل اللازمة لإدارة العقود والرواتب والتعديلات. سجل application_id محفوظ في trainee للرجوع للطلب الأصلي.

Phase 09 — بوابة المتدرب:
user account المُنشأ تلقائياً عند approve يُمكِّن المتدرب من تسجيل الدخول لبوابته.

Phase 07 — المخالفات:
المخالفات من درجة 5 (طرد) تُضيف إدخالاً في blocked_profiles تلقائياً بـ auto_blocked = true.

Phase 08 — الإشعارات:
national_id_expiry_date المحفوظ في trainees يُستخدم لإرسال تنبيه قبل 30 يوم.

نقاط التكامل الحرجة

registration_links.program_id → training_programs (Phase 02)

FK يجب أن يتحقق من وجود البرنامج قبل إنشاء الرابط. البرنامج يجب أن يكون is_active = true.

applications.approved → users (Phase 01)

إنشاء user في نفس transaction مع trainee. username = national_id، password = bcrypt(random_8_chars)، role = 'trainee'.

blocked_profiles.auto_blocked ← violations (Phase 07)

Phase 07 يكتب مباشرة في blocked_profiles عند تطبيق مخالفة درجة 5. هذا الاتجاه يعني Phase 03 يجب أن يبقى الـ schema مستقر.

trainees.national_id_expiry_date → notifications (Phase 08)

Phase 08 يقرأ هذا الحقل في cron job يومي للتحقق من البطاقات المنتهية قريباً.

المخاطر والافتراضات

ما يمكن أن يتأخر أو يُعقِّد التنفيذ

⚠️ المخاطر
  • 🔴Excel Parsing Complexity: ملفات Excel قد تأتي بـ encoding مختلف (Windows-1256 للعربية)، تنسيقات تاريخ متعددة (Hijri/Gregorian)، خلايا مدمجة، أو صفوف فارغة في الوسط. يجب استخدام مكتبة mature مثل exceljs مع معالجة شاملة للـ edge cases.
  • 🔴Storage Cost للصور: لو كل متدرب رفع صورة بـ 3-5MB، مع آلاف المتدربين التكلفة ترتفع. الحل: compress على الـ client قبل الرفع (max 400KB)، وتحويل لـ WebP. تحديد حد أقصى للحجم 2MB في backend أيضاً.
  • 🟡Spam عبر الروابط العامة: الروابط العامة مكشوفة — يمكن ملء النموذج بشكل آلي (bots). الحل: CAPTCHA (hCaptcha أو Google reCAPTCHA)، rate limiting per IP (max 3 submissions/hour)، fingerprinting الطلبات.
  • 🟡Race Condition في Capacity Check: لو قدَّم متدربان في نفس اللحظة وبقيت مقعد واحد، قد يُقبَلا معاً. الحل: استخدام SELECT FOR UPDATE أو transaction isolation level SERIALIZABLE على جدول groups.
  • 🟡Form Builder Backward Compatibility: لو عدَّل المسؤول بنية النموذج بعد وجود طلبات pending، قد تُقرأ form_data JSONB القديمة بشكل خاطئ. يجب تخزين snapshot من schema الحقول مع كل طلب.
📌 الافتراضات
  • ✅S3 bucket أو Minio instance مُعدَّة ومتاحة قبل بدء التطوير. التكوين (credentials, bucket name) محفوظ في .env.
  • ✅WhatsApp Business API (Twilio أو Meta Cloud API) متاح وله حصة كافية من الرسائل. نموذج رسالة approve و reject مُعتمَدان مسبقاً من Meta.
  • ✅SMTP مُعَدّ وعامل (SendGrid أو Mailgun) لإرسال رسائل القبول والرفض.
  • ✅training_programs و groups مُدخَّلة في النظام قبل بدء أي تسجيل. لا يمكن إنشاء رابط تسجيل لبرنامج غير موجود.
  • ✅الحقول الأساسية (الاسم، الهوية الوطنية، الجنس، تاريخ الميلاد، الجوال) موجودة دائماً في النموذج ولا يمكن حذفها من Form Builder — هي required hard-coded.
  • ✅الـ national_id يستخدم كـ unique identifier للمتدرب عبر كل النظام. لا يمكن وجود application أو trainee بنفس الـ national_id في نفس المجموعة.

هذه القائمة تُستخدَم كـ Definition of Done — المرحلة لا تنتهي حتى تُعلَّم جميع البنود بـ ✅.

  1. Backend — Form Builder: CRUD كامل لـ form_sections وـ form_fields مع validation على field_type وـ field_key uniqueness. Reordering endpoint يعمل بشكل atomic. ✅
  2. Backend — Registration Links: إنشاء links مع validation على expires_at و max_uses. endpoint عام GET /public/register/:code يتحقق من جميع قواعد الصلاحية (قاعدة 3.1) ويعيد HTTP 410 عند الفشل. ✅
  3. Backend — Applications: POST /public/applications يُشغِّل auto-review pipeline كاملاً (قواعد 3.2–3.5) ويحفظ النتيجة في auto_review_result. ✅
  4. Backend — Approve Transaction: POST /applications/:id/approve يُنفِّذ atomic transaction تنشئ users + trainees + تُحدِّث group.current_trainees + تُصدر BullMQ notification job (قاعدة 3.6). ✅
  5. Backend — Reject Validation: POST /applications/:id/reject يتحقق من rejection_reason (min 10 chars) في zod schema قبل تنفيذ أي شيء (قاعدة 3.9). ✅
  6. Backend — Bulk Import: POST /bulk-import يعالج .xlsx بـ exceljs، يطبق auto-review على كل صف، يعيد error report كـ Excel مع عمود "سبب الفشل" (قاعدة 3.8). ✅
  7. Backend — Blocked Profiles: CRUD لـ blocked_profiles. auto_blocked = true يُعيَّن فقط من Phase 07. رفع الحظر يُسجَّل في audit_logs. ✅
  8. Frontend — Form Builder UI: شاشة بناء النموذج تعمل بـ drag & drop للأقسام والحقول. معاينة حية للنموذج. لا يمكن حذف الحقول الإلزامية الأساسية (national_id، full_name، gender، dob، phone). ✅
  9. Frontend — Public Registration Page: تعمل على mobile (375px+). تعرض CAPTCHA. تضغط الصورة على الـ client. تعرض رسالة خطأ واضحة إذا كان الرابط منتهياً. تعرض رقم الطلب بعد الإرسال. ✅
  10. Frontend — Applications Review: قائمة مع فلاتر تعمل. auto_review_result يُعرَض بشكل مرئي واضح (pass/fail لكل check). Waitlist tab مستقل. ✅
  11. Frontend — Bulk Import: رفع ملف مع progress bar. معاينة أول 10 صفوف قبل Import. تنزيل error report بعد الانتهاء. ✅
  12. Database — Migrations: 6 جداول جديدة بـ migrations منفصلة. Indexes على: registration_links.code، applications.national_id، applications.status. ✅
  13. Testing — Auto-Review: unit tests لكل rule (3.1–3.5) مع edge cases: رابط منتهي بثانية واحدة، متقدم بعمر 17 سنة و364 يوم، مجموعة ممتلئة بالضبط. ✅
  14. Testing — Race Condition: integration test يُرسِل 2 طلبات متزامنة لمجموعة بمقعد واحد متبقٍّ — يتأكد من قبول طلب واحد فقط. ✅
